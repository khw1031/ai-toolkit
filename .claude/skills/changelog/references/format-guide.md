## 카테고리 순서

항상 아래 순서를 유지합니다 (해당 사항이 없는 카테고리는 생략):

1. `### ✨ 새 기능` — feat
2. `### 🐛 버그 수정` — fix
3. `### ♻️ 리팩토링` — refactor
4. `### 📝 문서` — docs
5. `### 💄 스타일` — style
6. `### ✅ 테스트` — test
7. `### 🔧 빌드/설정` — chore, build, ci
8. `### ⚠️ Breaking Changes` — BREAKING CHANGE

## 항목 작성 규칙

### 기본 형식

```
- 변경 내용 설명 — @담당자이름
```

### 상세 형식 (필요 시)

```
- 변경 내용 설명 — @담당자이름
  - 세부 변경 사항 1
  - 세부 변경 사항 2
```

### 담당자 표기

- `@이름` 형식으로 통일
- Git 커밋 author를 기본으로 사용
- 한 항목에 여러 담당자: `— @김철수, @이영희`
- 사용자가 담당자를 직접 지정할 수 있음

### 항목 설명

- 사용자 관점에서 작성 (기술 용어보다 기능 설명 우선)
- 동사로 시작: "추가", "수정", "개선", "삭제"
- 간결하게 한 줄로 요약 (필요 시 세부 항목 추가)

## 버전 헤더

```markdown
## [X.Y.Z] - YYYY-MM-DD
```

- 버전은 SemVer 형식
- 날짜는 ISO 8601 (`YYYY-MM-DD`)
- 버전과 날짜 사이에 ` - ` (공백-하이픈-공백)

## 커밋 → 카테고리 매핑

| 커밋 메시지 패턴 | 카테고리 |
|-----------------|----------|
| `feat:` / `feat(scope):` | ✨ 새 기능 |
| `fix:` / `fix(scope):` | 🐛 버그 수정 |
| `refactor:` | ♻️ 리팩토링 |
| `docs:` | 📝 문서 |
| `style:` | 💄 스타일 |
| `test:` | ✅ 테스트 |
| `chore:` / `build:` / `ci:` | 🔧 빌드/설정 |
| `BREAKING CHANGE` (footer) | ⚠️ Breaking Changes |
| 패턴 없음 | ✨ 새 기능 (기본값) |

## Conventional Commits 미사용 프로젝트

커밋 메시지에 타입 접두사가 없는 경우:

1. 커밋 메시지 내용으로 카테고리 추정
2. diff 내용 참고하여 분류
3. 불확실한 경우 사용자에게 확인
4. 기본값: ✨ 새 기능
