# Agent Skills 작성 규칙 (AgentSkills.io Specification)

> 스킬 생성, 수정, 검토 시 이 규칙을 따르세요.

## 디렉토리 구조

```
skill-name/
├── SKILL.md           # 필수 - 스킬 정의 파일
├── scripts/           # 선택 - 실행 가능한 코드
├── references/        # 선택 - 추가 문서
└── assets/            # 선택 - 정적 리소스
```

- `SKILL.md`는 반드시 스킬 디렉토리 루트에 위치
- 디렉토리명은 스킬의 `name` 필드와 일치

## SKILL.md 포맷

```yaml
---
name: skill-name          # 필수: 1-64자, 소문자/숫자/하이픈만
description: 스킬 설명     # 필수: 1-1024자, 무엇+언제
license: Apache-2.0       # 선택
compatibility: 환경 요구사항  # 선택: 1-500자
metadata:                  # 선택: 키-값 쌍
  author: org-name
  version: "1.0"
allowed-tools: Bash Read   # 선택, 실험적
---
```

## 필수 필드 규칙

### name
| 항목 | 규칙 |
|------|------|
| 길이 | 1-64 글자 |
| 허용 | 소문자, 숫자, 하이픈(`-`) |
| 금지 | 대문자, 시작/끝 하이픈, 연속 하이픈, 언더스코어 |

### description
- 1-1024 글자
- **무엇**을 하는지 + **언제** 사용하는지 명시
- 트리거 키워드 포함 (에이전트 스킬 선택 시 참조)

#### 트리거 키워드 작성 가이드

**효과적인 트리거를 위한 원칙:**

1. **자연어 형태로 작성** - 따옴표로 감싸지 않고 자연스럽게 나열
2. **구체적인 행동 동사 사용** - "프레임워크", "시스템" 같은 추상 명사 대신 동사 중심
3. **사용자 요청 패턴 반영** - 실제 사용자가 입력할 표현 사용
4. **다양한 표현 포함** - 같은 의도의 다른 표현들 나열

```yaml
# ❌ 비효과적 (트리거 실패 가능)
description: |
  코드 리뷰를 위한 프레임워크.
  사용 시점: "코드 리뷰", "리뷰 요청"

# ✅ 효과적 (트리거 성공률 높음)
description: |
  코드 변경사항을 검토하고 피드백을 제공합니다.
  코드 리뷰, 리뷰해줘, PR 검토, 코드 검토, 변경사항 확인 시 사용하세요.
```

**기본 패턴:**
```
[무엇을 하는지 설명합니다.]
[키워드1], [키워드2], [키워드3] 시 사용하세요.
```

#### Description 구성 패턴

스킬이 여러 상황에서 트리거되어야 한다면, 상황별로 섹션을 나누고 각각의 트리거 패턴을 명시하세요.

**패턴 1: 상황별 섹션 분리**
```yaml
description: |
  [스킬의 전체 목적 설명]

  [상황 1] 시:
  - [키워드 그룹 1]
  - [영어 키워드들]

  [상황 2] 시:
  - [키워드 그룹 2]
  - [실제 사용 예시]
```

**예시: 워크플로우 스킬**
```yaml
description: |
  일반적인 기능 구현을 위한 단계별 워크플로우를 제공합니다.

  새 기능 구현 시:
  - 기능 구현, 새 기능, 기능 추가, 신규 개발
  - feature implementation, new feature, add feature

  작업 재개 시 (티켓 번호 패턴 포함):
  - "PROJ-001 작업 이어서", "FEAT-123 계속", "TASK-456 진행"
  - 작업 이어서, 작업 계속, 이어서 진행
  - resume task, continue work, proceed
```

**패턴 2: 조합 패턴 명시**

실제 사용자 입력에 특정 조합이 있다면 예시로 명시:
```yaml
description: |
  [기능 설명]

  트리거 패턴:
  - "[식별자] + [동작]": "PROJ-001 시작", "TASK-123 계속"
  - "[컨텍스트] + [동작]": ".ai/tasks/ 작업 진행"
```

**패턴 3: 다국어 키워드**

한글/영어 사용자 모두 지원:
```yaml
description: |
  [기능 설명]

  - 한글: 키워드1, 키워드2, 키워드3
  - English: keyword1, keyword2, keyword3
```

**패턴 4: 부정 패턴 (선택)**

스킬이 트리거되지 말아야 할 경우를 명시:
```yaml
description: |
  [기능 설명]
  [트리거 키워드들]

  주의: "목록 보기", "상태 확인"만 요청 시에는 사용하지 않습니다.
```

## 컨텍스트 효율성 (Progressive Disclosure)

```
1단계: 메타데이터 (~100 토큰)
    → name, description (시작 시 항상 로드)

2단계: 지침 (<5000 토큰 권장)
    → SKILL.md 전체 본문 (스킬 활성화 시)

3단계: 리소스 (필요시)
    → scripts/, references/, assets/ (온디맨드)
```

**IMPORTANT**: 본문 500줄 이하 유지, 상세 내용은 `references/`로 분리

## 파일 참조

- 스킬 루트에서 상대 경로 사용
- 1단계 깊이 유지 (깊은 중첩 피하기)

```markdown
# ✅ 올바른 참조
[상세 가이드](references/GUIDE.md)

# ❌ 피해야 할 참조
[문서](references/sub/deep/nested/doc.md)
```

## scripts/ 작성 원칙

1. 자체 포함 또는 의존성 명확히 문서화
2. 도움이 되는 에러 메시지 제공
3. Usage 문서화 필수

## 검증 체크리스트

```
□ name: 1-64자, 소문자/숫자/하이픈만, 디렉토리명과 일치
□ description: 1-1024자, 무엇+언제 포함
□ description: 트리거 키워드 자연어 형태 (따옴표 없음)
□ description: 구체적 행동 동사 사용 (추상 명사 지양)
□ SKILL.md: < 500줄
□ 파일 참조: 1단계 깊이
□ scripts/: 실행 가능, 에러 처리 포함
```

## 안티패턴

- 모호한 description (예: "유용한 도구입니다")
- 따옴표로 감싼 트리거 키워드 (예: `"코드 리뷰"`, `"PR 검토"`)
- 추상적 명사 중심 설명 (예: "프레임워크", "시스템", "도구")
- 500줄 초과 SKILL.md
- 하드코딩된 절대 경로
- 외부 의존성 미문서화
- 깊은 참조 체인
